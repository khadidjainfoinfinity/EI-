Examen — Sujet A (10 questions)

USTHB / L3 ACAD — chaque question est suivie de sa solution corrigée.

Q1 — XQuery erreur d'attribut

Soit : element racine {attribute lang {"fr"}, attribute lang {"en"}}
1) Pourquoi génère-t-elle une erreur ?   2) Comment la corriger ?

1) Erreur : un élément ne peut pas avoir deux attributs portant le même nom (lang est déclaré deux fois).
2) Corriger en renommant l'un d'eux, ou en fusionnant les valeurs :
element racine {attribute lang {"fr"}, attribute langue {"en"}}
ou element racine {attribute lang {"fr en"}}
Q2 — XQuery, texte non quoté

Soit : element racine {attribute lang {"fr"}, Texte}
1) Pourquoi génère-t-elle une erreur ?   2) Comment la corriger ?

1) Texte est écrit sans guillemets : il est donc interprété comme une expression de chemin XPath (recherche d'un élément nommé Texte dans le nœud de contexte). Comme aucun contexte n'est défini → erreur (« context item is absent / no context value »).
2) Mettre des guillemets pour en faire un littéral de chaîne :
element racine {attribute lang {"fr"}, "Texte"} → <racine lang="fr">Texte</racine>
Q3 — XPath : . vs text() (avec sous-élément)

XML : <a who="C" n="9">donc <vocal desc="e"/> ben je prends 17h quand même</a>

Expliquer : //a[contains(., 'quand')]  et  //a[contains(text(), 'quand')]

//a[contains(., 'quand')] → . = tout le contenu textuel de a (« donc … quand même », y compris le texte après <vocal/>). « quand » s'y trouve → retourne l'élément <a>.
//a[contains(text(), 'quand')] → text() = seulement le premier nœud de texte direct (« donc »). « quand » se trouve après <vocal/>, dans un autre nœud texte → ne retourne rien.
Q4 — XPath : . vs text() (texte simple)

XML : <u><a>quand</a></u>

Expliquer : //a[text()="quand"]  et  //a[.="quand"]

Ici a ne contient qu'un seul nœud de texte direct (« quand »), sans sous-élément. Donc :
text() = « quand » et . (valeur textuelle de a) = « quand ».
Les deux expressions donnent le même résultat → retournent <a>quand</a>.
⟶ Contrairement à Q3, il n'y a pas de sous-élément qui « coupe » le texte, donc . et text() coïncident.
Q5 — Expliquer une requête FLWOR (jointure)
for $b1 in doc("books1.xml")//book, $b2 in doc("books2.xml")//book
where $b1/@* = $b2/@year
let $both := ($b2/*/*[2], $b1/*/*[2])
for $ele at $i in $b2/*/*[2]
return <a>{$ele, $both[$i + count($both) div 2]}</a>
  • Double for = jointure : on apparie chaque book de books1 avec chaque book de books2.
  • where $b1/@* = $b2/@year : on ne garde que les paires où un attribut de $b1 est égal à l'attribut year de $b2.
  • let $both : séquence qui regroupe le 2ᵉ sous-sous-élément de $b2 puis celui de $b1 (les deux moitiés).
  • for $ele at $i : on itère sur les 2ᵉ sous-sous-éléments de $b2 ; at $i = compteur de position.
  • return <a>{$ele, $both[$i + count($both) div 2]}</a> : pour chaque $ele, on l'apparie avec l'élément de $both situé à la position $i + (moitié de la longueur de $both) → on associe ainsi chaque élément de la 1ʳᵉ moitié de la séquence à son correspondant dans la 2ᵉ moitié, dans un élément <a>.
Q6 — XQuery : axe attribute, parent::, racine

XML : <arbres><arbre id="6"><nom_commun>Oranger des osages</nom_commun></arbre>…</arbres>

R1 : for $arbre in doc("arbres.xml")//arbre
     return element nom{$arbre/nom_commun/text()," ID",string($arbre/attribute::id)}

R2 : for $nom in doc("arbres.xml")//nom_commun
     return element nom{$nom/....," ID",string($nom/....::....)}
1) Dans R1, l'axe attribute::id peut être remplacé par @id (@ est le raccourci de attribute::).
2) Compléter R2 (afficher le nom commun + son ID, alors que l'id est sur le parent arbre) :
return element nom{$nom/text()," ID",string($nom/parent::arbre/@id)} (on remonte de nom_commun vers son parent arbre pour atteindre @id ; parent::arbre ≡ ..)
3) Si on ajoute id="6" à l'élément racine <arbres> : les deux requêtes continuent de fonctionner. R1 lit //arbre/@id (l'id de chaque arbre) et R2 lit parent::arbre/@id (l'id du parent arbre). La racine <arbres> n'est jamais sélectionnée comme un arbre, donc son nouvel id est ignoré. R2 est la plus robuste car elle ancre explicitement sur parent::arbre.
Q7 — XPath : 3 expressions de position

Expliquer la différence : /descendant::titleStmt[2] , (//titleStmt)[2] , //titleStmt[2]

/descendant::titleStmt[2] ≡ //titleStmt[2] → « le titleStmt qui est en 2ᵉ position chez son parent » → peut donner 0, 1 ou plusieurs résultats.
(//titleStmt)[2] → on collecte tous les titleStmt puis on prend le 2ᵉ de la liste globale → toujours 1 seul résultat.
Q8 — Code JAVA/DOM (texte parasite + filtrage)
<liste>
L2<eleve nom="E1" note="12">L2 ACAD</eleve>
L3<eleve nom="E2" note="18">L3 ACAD</eleve>
L2<eleve nom="E3" note="07">L2 ACAD</eleve>
</liste>
NodeList etudiant = document.getDocumentElement().getChildNodes();
for (int o = 0; o<etudiant.getLength(); o++){
  Filtrer_parasites a = new Filtrer_parasites();   // Filtrer renvoie (getNodeName() != "#text")
  if(a.Filtrer(etudiant.item(o)))
     System.out.println(etudiant.item(o).getFirstChild().getNodeValue());
  else
     if (!etudiant.item(o).getNodeValue().endsWith( "…." ))     // ← ligne en gras
        System.out.println(etudiant.item(o).getNodeValue().replaceAll("\\n|\\t",""));
}
1) Compléter la ligne en gras : endsWith("\n").
Les fils de <liste> sont les #text « \nL2 », « \nL3 », « \nL2 » (utiles) et le dernier « \n » (parasite). Seul le parasite se termine par « \n ».

2) Ce que le code affiche (les eleve → leur texte ; les #text utiles → valeur sans \n/\t ; le #text « \n » → ignoré) :
L2 L2 ACAD L3 L3 ACAD L2 L2 ACAD
3) Rôle de la ligne en gras : elle écarte le nœud #text parasite qui n'est qu'un saut de ligne (le dernier « \n » avant </liste>), pour ne pas afficher de ligne vide ; elle distingue les vrais textes (« L2/L3 ») des blancs parasites.

4) Remplacer \n|\t : par \s (classe « espace blanc » qui couvre \n, \t, espace…) → même résultat ici. (équivalent exact : [\n\t])
Q9 — DTD : attribut ID invalide

XML : <personne datenaissance="a 1942-01-08"/>
DTD : <!ATTLIST personne datenaissance ID #REQUIRED>

1) Erreur : un attribut de type ID ne peut pas contenir d'espace (ni commencer par un chiffre). La valeur « a 1942-01-08 » contient un espace.
2) Deux corrections possibles :
  • (a) changer le type en CDATA : <!ATTLIST personne datenaissance CDATA #REQUIRED>
  • (b) enlever l'espace dans la valeur : datenaissance="a1942-01-08"
Q10 — DTD : contenu mixte

XML : <name>W.B<family>Wallace</family></name>
DTD : <!ELEMENT name (#PCDATA, family)>

Erreur : name a un contenu mixte (le texte « W.B » + le sous-élément family). On ne peut pas écrire (#PCDATA, family).
Correction : <!ELEMENT name (#PCDATA | family)*> (forme obligatoire du contenu mixte).

Q2 — Que retourne l'expression taggertoken ?
//taggertoken[contains(@pos,'ADJ')]/following-sibling::taggertoken[1][contains(@pos,'NOM')] |
//taggertoken[contains(@pos,'NOM')]/following-sibling::taggertoken[1][contains(@pos,'ADJ')] |
//taggertoken[contains(@pos,'NOM')]/preceding-sibling::taggertoken[1][contains(@pos,'ADJ')] |
//taggertoken[contains(@pos,'ADJ')]/preceding-sibling::taggertoken[1][contains(@pos,'NOM')] | …
Cette expression recherche les séquences Nom + Adjectif ou Adjectif + Nom adjacentes (un mot dont le voisin immédiat a le POS complémentaire).
  • following-sibling::taggertoken[1] = le frère immédiatement suivant ; preceding-sibling::…[1] = le frère immédiatement précédent.
  • [contains(@pos,'…')] teste l'étiquette grammaticale (ADJ / NOM).
  • L'union | combine tous les cas : ADJ suivi d'un NOM, NOM suivi d'un ADJ, et leurs symétriques avec preceding.
Résultat : tous les taggertoken qui forment une paire nom + adjectif contiguë (dans les deux ordres). C'est l'exercice « rechercher les séquences NOM ADJ ou ADJ NOM ».
Q3 — XQuery : deux attributs de noms différents

Soit : element racine {attribute lang_fr {"fr"}, attribute lang_en {"en"}}
1) Génère-t-elle une erreur ?   2) Si oui, corrigez-la.

1) Non, pas d'erreur : les deux attributs ont des noms différents (lang_fr et lang_en), ce qui est autorisé. (Contraste avec le Sujet A Q1 où les deux s'appelaient lang → erreur.)
Résultat : <racine lang_fr="fr" lang_en="en"/>
2) Rien à corriger. ⚠️ Veiller à écrire le mot-clé attribute en minuscule (XQuery est sensible à la casse) ; Attribute avec majuscule serait une erreur de syntaxe.
Q4 — XQuery : opérateur is

XML : <element><a>5</a><b>5</b><a>5</a></element>
Requête : doc("fic1.xml")//a[1] is doc("fic1.xml")//a[2]

Résultat : false.
is compare l'identité des nœuds (pas leur valeur). Les deux <a> sont deux nœuds différents, chacun avec sa propre identité, même s'ils ont la même valeur (5). Donc false.
Q5 — Expliquer une construction element/attribute
element { doc("books2.xml")//book[1]/name(@*) } {
  attribute { doc("books2.xml")//book[1]/name(*[3]) }
            { doc("books2.xml")//book[1]/*[3] }
}

books2.xml : <b><book title="XQuery"><author>…</author><publisher>O'Reilly</publisher></book><book/></b>

  • Nom de l'élément = name(@*) de book[1] : ce premier book a un seul attribut (title) → name(@*) = « title » → l'élément s'appellera <title>.
  • Attribut interne : son nom = name(*[3]) (nom du 3ᵉ élément-enfant de book[1]) et sa valeur = *[3] (contenu de ce 3ᵉ enfant).
⚠️ Problème : book[1] n'a que 2 éléments-enfants (author et publisher). Donc *[3] = séquence vide → name(*[3]) = « » (chaîne vide) → on construit un attribut sans nom → ERREUR (nom d'attribut vide invalide).
⟶ Si c'était *[2] : name(*[2])=« publisher », *[2]=« O'Reilly » → <title publisher="O'Reilly"/>.
Q6 — encoding obligatoire ? + requête analyze-string
<?xml version="1.0" encoding="iso-8859-1"?>
<classification_atomique type="gaz rare">
  <atome><nom>hélium</nom><masse>la masse est 4</masse></atome>
  <atome><nom>neon</nom><masse>20</masse></atome>
  <atome><nom>argon</nom><masse>40</masse></atome>
</classification_atomique>
for $i in doc("fic2.xml")//masse
where analyze-string($i,"\d+")//fn:match < 20
return $i
1) L'attribut encoding est-il obligatoire ?
En général il est facultatif (l'encodage par défaut est UTF-8/UTF-16). Mais ici, oui, il est nécessaire : le document est encodé en iso-8859-1 (qui n'est pas l'encodage par défaut) et contient des caractères accentués (« hélium »). Sans la déclaration encoding, le parseur supposerait UTF-8 et mal interpréterait les accents (caractères corrompus / erreur). Il faut donc préciser encoding="iso-8859-1".

2) Explication de la requête :
  • analyze-string($i, "\d+") applique la regex \d+ (un ou plusieurs chiffres) sur le texte de chaque masse et renvoie une structure avec des <fn:match> (les chiffres trouvés).
  • //fn:match récupère le nombre extrait de chaque masse ; < 20 garde celles dont le nombre est strictement inférieur à 20.
  • Valeurs : « la masse est 4 » → 4 < 20 = vrai ✔ · « 20 » → 20 < 20 = faux ✘ · « 40 » → faux ✘.
Résultat : <masse>la masse est 4</masse>
Q7 — Code JAVA/DOM récursif (erreur de type)

XML : <liste><eleve nom="Et2" note="15">L2 ACAD<s>A<u>C</u></s></eleve></liste>

public static void L(Node n){
  for (int o=0; o<n.getChildNodes().getLength(); o++){
     Element z = n.getChildNodes().item(o);     // ← ligne fautive
     if(z.getNodeName()=="#text")
        System.out.println(z.getNodeValue());
     L(z);
  }
}
1) Pourquoi une erreur ? item(o) renvoie un Node, pas un Element. On ne peut pas affecter directement un Node à une variable Element sans conversion → erreur de compilation (« Type mismatch: cannot convert from Node to Element »). De plus, un nœud #text n'est pas un Element (un cast forcé lèverait ClassCastException à l'exécution).

2) Correction : déclarer z comme Node (on n'utilise que getNodeName()/getNodeValue(), méthodes de Node) :
Node z = n.getChildNodes().item(o);

3) Ce qu'affiche le code (parcours récursif de tout l'arbre, affichant la valeur de chaque #text). Textes présents : « L2 ACAD » (dans eleve), « A » (dans s), « C » (dans u) — le XML est sur une seule ligne, donc pas de blancs parasites :
L2 ACAD A C
Justification : L() descend récursivement dans eleve → s → u ; à chaque nœud #text rencontré, il imprime sa valeur (dans l'ordre du document).
